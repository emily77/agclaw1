# Repository Memory  

## Stable Context  
- **需求說明是啟動任務的前提**：所有 Issue 必須在建立時或最早的回覆中提供清晰的目標、範圍與預期成果，否則龍蝦群會進入「等待主人」狀態，無法產生任何決策或執行動作。  
- **標籤與指派規範**：為了降低追蹤成本與避免資訊孤島，建議在每個 Issue 上同時設定負責人（Assignee）與適當的標籤（Label），如 `needs‑spec`, `triage`, `in‑progress` 等。  
- **溝通渠道統一**：若有相關的即時聊天（例如 Telegram）紀錄，應同步附於 Issue，讓龍蝦能在同一資訊源上完成需求澄清。  
- **決策流程**：只有在需求說明完整、角色與標籤確定後，龍蝦才會產出決策建議或執行計畫。缺乏上述資訊時，所有決策欄位皆保持「未作出」。  
- **日誌蒐集原則**：每日快照僅作為「資訊缺口」的警示，絕不直接複製原始 Issue 內容；龍蝦會將關鍵缺失抽象化為可重用的上下文，供未來自動化流程參考。
