# Repository Memory

## Stable Context
- **專案治理**  
  - 所有工作皆以 GitHub Issue 為單位，並透過 **compact‑memory workflow** 讓龍蝦 agents 每日抓取快照。  
  - `shared/manual.md` 為唯一由人類手動維護的長期記憶檔，僅存放 **穩定規則、長期決策、常見限制與 repo 習慣**，不會被自動覆寫。  

- **Issue 建立原則（建議）**  
  1. **需求說明必填**：標題須簡潔，內容必須包含目標、預期成果、相關背景或對話紀錄（如 Telegram）。  
  2. **指派與標籤**：建立 Issue 時即指定負責人並加上適當標籤，以降低後續追蹤成本。  
  3. **文件化**：任何需求變更或決策應以文字文件形式保存，供龍蝦 agents 參考。  

- **龍蝦 Agents 的角色**  
  - 每日掃描所有 Issue，彙整「Agent Activity」與「Cross‑Issue Themes」。  
  - 只在資訊足夠時才產出決策或行動建議，資訊不足則回報 **Open Loops**。  

- **資訊缺口的處理流程**  
  1. 發現缺少需求說明 → 在每日快照中標記為 **Open Loop**。  
  2. 主人提供完整說明 → 龍蝦重新評估，產出決策與待辦。
