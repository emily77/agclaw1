# Repository Memory

## Stable Context
- **Repository**: `emily77/agclaw1`  
- **Issue Management**  
  - 每個 Issue 必須在建立時提供 **完整的標題、描述、目標與預期成果**，否則龍蝦們無法展開工作。  
  - 需求說明應包含 **功能需求、文件撰寫、測試計畫** 等可執行的子任務。  
  - 建議在 Issue 建立時即指派負責人（agent）並加上適當的 **Label**（如 `feature`, `bug`, `documentation`），以降低後續追蹤成本。  
- **Agent 工作流程**  
  1. **資訊收集**：先檢視 Issue 是否具備足夠資訊。  
  2. **需求確認**：若資訊不足，標記為 *Open Loop* 並等待主人回覆。  
  3. **分配任務**：資訊完整後，依需求範圍分配負責的龍蝦（agent）。  
  4. **設定里程碑**：為每個子任務設定交付期限與驗收標準。  
- **限制**  
  - 每次快照僅檢視最近 30 天內、**最多 100 個 Issue**。  
  - 只有 **GitHub Issue / Comment** 被視為原始資料來源，手動筆記不會被覆寫。  
- **長期決策**  
  - 建立一套 **需求收集與文件化流程**（例如：在 Issue 建立時附上需求表單），以避免持續的資訊缺口。  
  - 讓所有龍蝦遵守「資訊完整即行動」的原則：只要 Issue 具備完整描述，即可自動觸發分配與里程碑設定。
